## Context

The Release workflow (`.github/workflows/release.yml`) triggers on push to `main` (stable) and `develop` (prerelease). It builds platform binaries, then a `release` job creates a GitHub Release via `softprops/action-gh-release`. Two problems exist today:

1. **Tag commitish defaults to the repository default branch.** `softprops/action-gh-release` defaults `target_commitish` to the repository default branch (main), not `github.sha`. So a develop push produces a prerelease tag pointing at main's HEAD, not the develop commit that was pushed. The tag and the built artifacts (which were compiled from the develop checkout) become inconsistent.
2. **Changelog is mechanical.** The changelog step runs `git log --pretty=format:"%s"` and greps conventional-commit prefixes. This loses the semantic context of what actually changed in the byok program and its specs.

The user wants the changelog generated by an AI model using the GitHub Models free tier, reading the changed byok source code and specification documents in the release range.

## Goals / Non-Goals

**Goals:**

- Every release tag SHALL point at the commit that triggered the workflow (`github.sha`), so develop prereleases land on develop commits and main stable releases land on main commits.
- The changelog SHALL be generated by a GitHub Models model call that consumes the diff of byok-relevant files (`cmd/`, `internal/`, `openspec/specs/`) in the release range and produces a categorized Markdown changelog (新增功能 / 優化功能 / 修復功能).
- A deterministic fallback SHALL keep releases working when the model call is unavailable or fails.

**Non-Goals:**

- No change to version numbering, build matrix, artifact naming, or the Pushover notify job.
- No change to the `byok` application code or CLI behavior.
- No paid external LLM service; only GitHub Models free tier.
- No regeneration of historical release changelogs.

## Decisions

### Decision: Set `target_commitish` to `github.sha` on the release action

Set `target_commitish: ${{ github.sha }}` on the `softprops/action-gh-release` step in the `release` job. `github.sha` is the commit that triggered the workflow run, which is the develop commit for a develop push and the main commit for a main push. This is the minimal fix and requires no new permissions.

**Alternative considered:** Compute the branch name and pass `target_commitish: ${{ github.ref_name }}`. Rejected because a branch name commitish resolves to the branch HEAD at tag-creation time, which can race with subsequent pushes; `github.sha` pins the exact commit that was built and tested.

### Decision: Generate the changelog via GitHub Models using a release-ranged diff prompt

Add a `Generate changelog` step (replacing the current grep step as the primary path) that:

1. Resolves the previous release tag with `git describe --tags --abbrev=0` (or empty for the first release).
2. Computes the diff of byok-relevant paths (`cmd/`, `internal/`, `openspec/specs/`) between the previous tag and `HEAD` via `git diff`.
3. Calls a GitHub Models model endpoint using the workflow's `GITHUB_TOKEN` (or a dedicated Models token secret) with a prompt instructing it to produce a Markdown changelog categorized into 新增功能 / 優化功能 / 修復功能, describing the byok program changes in the provided diff.
4. Writes the model output to `changelog.md`.

**Alternative considered:** Use the GitHub-hosted `@github/models` action wrapper. Rejected for portability; a direct `curl` to the GitHub Models inference endpoint keeps the workflow self-contained and explicit, and the prompt/diff assembly is auditable in the workflow file.

### Decision: Retain a commit-history fallback step

If the model call exits non-zero, returns an empty body, or the Models token is unset, run the existing `git log` + grep categorization as a fallback and write that to `changelog.md`. The release step then uses whatever `changelog.md` contains. This guarantees a release is never blocked by an AI dependency.

### Decision: Scope the diff sent to the model to byok-relevant paths

The prompt includes `git diff` output restricted to `cmd/ internal/ openspec/specs/`. Workflow and doc-only changes (e.g. `.github/workflows/`, `README.md`, `AGENTS.md`) are summarized at a high level in the prompt context rather than sent verbatim, to keep the prompt within model context limits and keep the changelog focused on byok program behavior.

## Implementation Contract

- **Behavior (operator-visible):** After this change, a push to `develop` creates a prerelease whose tag points at the develop commit (verifiable in the GitHub release UI: the tag's commit SHA equals the pushed commit), and a push to `main` creates a stable release tagged on the main commit. The release body is an AI-generated categorized changelog describing byok source/spec changes, or a commit-history categorized changelog when the model is unavailable.
- **Interface / data shape:** The `release` job's `softprops/action-gh-release` step receives an explicit `target_commitish: ${{ github.sha }}` input. The `Generate changelog` step produces a `changelog.md` file at the workflow workspace root, consumed by `body_path: changelog.md`. The GitHub Models call uses the Models inference HTTP endpoint with a JSON body containing the model id, a system prompt defining the categorization format, and a user message containing the assembled diff.
- **Failure modes:** Model HTTP error, timeout, empty response, or unset token → fallback commit-history changelog is written and the release proceeds. Tag commitish fix has no failure mode beyond the release action's own behavior.
- **Acceptance criteria:**
  - On a develop push, the created release tag's commit SHA matches the pushed develop commit SHA (not the main branch HEAD).
  - On a main push, the created release tag's commit SHA matches the pushed main commit SHA.
  - When the model call succeeds, the release body contains 新增功能/優化功能/修復功能 sections derived from the byok diff.
  - When the model call is forced to fail (e.g. unset token), the release still succeeds with a commit-history categorized changelog body.
- **Scope boundaries:**
  - In scope: `.github/workflows/release.yml` `release` job (tag commitish + changelog steps).
  - Out of scope: build job, notify job, version numbering, application code, other workflows.

## Risks / Trade-offs

- **Model output non-determinism** → The changelog content varies run to run. Mitigation: pin the model id and keep the prompt stable; the fallback ensures a release always has a body.
- **GitHub Models free-tier rate limits** → Concurrent develop pushes could hit limits. Mitigation: the fallback keeps releases working; limits are unlikely for this repo's push cadence.
- **Large diffs exceed model context** → Restricting the diff to `cmd/ internal/ openspec/specs/` bounds the size; very large changes can still be big. Mitigation: summarize workflow/doc-only changes rather than sending them verbatim, and truncate per-file hunks if needed.
- **Token leakage** → The diff may contain secrets if one is committed. Mitigation: the diff is byok source/spec only; secrets live in keychain/GitHub secrets, not in these paths.
